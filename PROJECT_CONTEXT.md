# 项目上下文

## 0. 目的

本仓库是一个数学建模工作流项目。

目标是为数学建模竞赛和研究式建模任务建立一套可复现、可审计、合同驱动的工作流。

工作流应支持：

1. 赛题接收与任务拆解；
2. 模型族路由；
3. 在不复制历史论文的前提下检索先验经验卡片；
4. 基于代码求解并冻结结果；
5. 将图表绑定到结果和论断；
6. 基于证据生成论文草稿；
7. 多审稿器质量审查；
8. 修订任务追踪；
9. 最终润色和导出。

项目优化目标：

- 正确性；
- 可复现性；
- 可追踪性；
- 人类控制；
- Web GPT、Codex 与本地 DeepSeek Claude 之间低摩擦协作。

## 2. 核心原则

### 2.1 合同优先

所有重要输出最终都必须进入合同文件。

重要合同包括：

```text
14_contracts/result_contract.csv
14_contracts/figure_contract.csv
14_contracts/formula_contract.csv
14_contracts/claim_evidence_map.csv
14_contracts/citation_contract.csv
```

生成段落、图表或结论，只有能追踪到合同记录时才可信。

### 2.2 代码输出是最高事实来源

事实层级：

```text
1. 原始数据和已执行代码输出
2. 冻结的 result_contract 行
3. 由冻结结果生成的图表文件
4. 论断-证据映射
5. 论文草稿文本
6. AI 评论和建议
```

论文文本不得覆盖代码输出。

如果论文文本与 `result_contract` 冲突，以 `result_contract` 为准。

### 2.3 不盲目堆功能

新功能只有具备以下工程链路才可进入核心工作流：

```text
问题
→ 原因
→ 设计变化
→ 受影响文件
→ 校验方法
→ 回滚方法
→ 风险控制
```

没有校验方式的功能不得进入核心工作流。

### 2.4 关键闸门必须有人参与

以下事项必须人工确认：

1. 最终模型族选择；
2. 重要数值结果冻结；
3. 图表进入论文；
4. 接受 major 级审稿修订；
5. 最终论文导出。

工作流可以建议，但不得静默决定这些事项。

### 2.5 AI 不得绕过闸门

AI 代理不得：

- 修改 `workflow_state.yaml` 来跳阶段；
- 在结果冻结前写最终结论；
- 在未验证引用时写引用支撑句；
- 用润色改变事实；
- 删除日志、合同或审稿记录来让检查通过。

## 3. 角色边界

### 3.1 Web GPT

主要负责：

- 架构设计；
- 路线规划；
- 审查和复盘；
- 提示词设计；
- 最终决策支持。

### 3.2 Codex

主要负责：

- 精准实现；
- 调试；
- 本地校验；
- 变更说明；
- 受控文件编辑。

### 3.3 本地 DeepSeek Claude

主要负责：

- 批量扫描；
- 摘要整理；
- 草稿候选；
- 训练沙盒中的低成本实验。

它不得独立决定核心架构，也不得绕过正式工作流。

## 6. 工作方式

任何任务都应尽量通过文件交接：

- `TASK_PACKET.md` 描述当前任务范围；
- `DECISIONS.md` 记录长期架构决策；
- `CHANGELOG_AI.md` 记录 AI 辅助变更；
- `PROJECT_CONTEXT.md` 提供项目地图；
- `AGENTS.md` 约束代理行为。

如果任务会改变正式流程，必须说明：

```text
改什么
为什么改
影响哪些文件
如何验证
如何回滚
有哪些风险
```

---

## 7. 非目标

本项目不是：

- 一键生成获奖论文的黑箱；
- 历史论文复用器；
- 自动跳过人类确认的流水线；
- 无合同约束的自由写作代理；
- 多代理堆叠实验场。

---

## 8. 当前安全基线

正式工作流必须满足：

```text
execution_mode: deep_sequential
allow_parallel: false
```

关键提交前必须运行：

```bash
python scripts/validate_contracts.py --stage current
python scripts/check_gates.py
```

最终导出前还必须确认：

- 没有未关闭 fail 级审稿问题；
- 必需修订任务已关闭或人工豁免；
- 润色没有改变受保护事实；
- 图表、结果、公式、引用和论断都能追踪到合同；
- 人类最终闸门已确认。
