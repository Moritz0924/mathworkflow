## ADR-001：保持 Web GPT 负责架构，Codex 负责实现

#### 状态

```text
accepted
```

#### 日期

```text
YYYY-MM-DD
```

#### 背景

项目使用三类 AI 资源：

```text
1. Web GPT Team：推理强，主要用于架构和复盘。
2. Codex：有本地工程能力，但使用时间有限。
3. 本地 DeepSeek Claude：成本低，但项目治理能力较弱。
```

如果三者混用，会造成交互混乱、重复推理和项目状态不一致。

#### 决策

采用以下角色划分：

```text
Web GPT Team:
- 架构
- 规划
- 审查
- 提示词设计
- 最终决策支持

Codex:
- 精准实现
- 调试
- 校验
- diff 报告

Local DeepSeek Claude:
- 扫描
- 摘要
- 草稿生成
- 批量整理
```

#### 推理

Codex 时间有限，因此不应用于宽泛分析或长时间规划。

Web GPT 更适合全局推理，但缺少本地项目直接访问能力。

本地 DeepSeek 成本低，适合批量阅读，但不应独立做核心架构决策。

该分工能降低成本和协作失败率。

#### 后果

正面：

```text
- Codex 时间保留给实现。
- Web GPT 继续负责设计一致性。
- 本地 DeepSeek 可承担低成本上下文抽取。
```

负面或权衡：

```text
- 需要严格的文件式交接。
- 步骤之间需要人工审查。
```

#### 相关文件

```text
- PROJECT_CONTEXT.md
- TASK_PACKET.md
- CHANGELOG_AI.md
```

```text
P0
```

## ADR-002：使用合同驱动工作流，而不是自由形式代理输出

#### 状态

```text
accepted
```

#### 背景

数学建模工作流容易出现虚构数字、无支撑论断、图表不一致和论文文本偏离代码输出的问题。

#### 决策

使用合同作为中央控制总线。

重要生成产物必须能通过合同文件追踪。

关键合同包括：

```text
result_contract.csv
figure_contract.csv
formula_contract.csv
claim_evidence_map.csv
citation_contract.csv
```

#### 推理

自由形式代理输出很难校验。

合同行可以被检查、引用、比较、冻结和审查。

这使后续校验脚本和闸门检查成为可能。

#### 后果

正面：

```text
- 数值论断可追踪。
- 图表使用可检查。
- 论文章节可绑定证据。
```

负面或权衡：

```text
- 前期结构工作更多。
- 代理必须遵循更严格输出格式。
```

#### 相关文件

```text
- 14_contracts/
- scripts/validate_contracts.py
- scripts/check_gates.py
```

```text
P4
P5
P6
```

## ADR-003：在关闭审稿闭环前，不增加更多核心代理

#### 状态

```text
accepted
```

#### 背景

一些开源数学建模代理使用大量专门代理。本项目已有审稿角色，但“审稿到修订关闭”尚未完全可执行。

#### 决策

核心审稿器保持为：

```text
problem_reviewer
model_reviewer
code_reviewer
figure_reviewer
paper_reviewer
judge_reviewer
```

在修订任务生成和关闭检查实现前，不增加更多核心代理。

#### 推理

当前瓶颈不是代理数量，而是：

```text
审稿意见
→ 结构化问题
→ 修订任务
→ 修复
→ 关闭验证
```

更多代理会产生更多评论，但不一定更可执行。

#### 后果

正面：

```text
- 编排复杂度更低。
- 更容易校验。
- 审稿职责更清晰。
```

负面或权衡：

```text
- 部分专门检查会延后。
```

## ADR-004：先建立阶段提示词合同，再建立技能路由

#### 状态

```text
accepted
```

#### 背景

项目计划增强技能集成，但技能权限依赖工作流阶段边界。

如果没有标准化阶段提示词，Skill Router 无法知道每个阶段允许哪些文件、合同和动作。

#### 决策

先实现 P0 阶段提示词合同，再实现 P1 Skill Router。

#### 推理

技能路由需要：

```text
当前阶段
允许读取路径
允许写入路径
必需输入
预期输出
禁止动作
校验命令
```

这些属于阶段提示词合同。

#### 后果

正面：

```text
- Skill Router 可复用阶段权限。
- 后续校验更简单。
- 代理行为更可预测。
```

负面或权衡：

```text
- 技能集成会推迟到提示词标准化之后。
```

## ADR-005：Prior DB 必须产出卡片，而不是论文文本

#### 状态

```text
accepted
```

#### 背景

历史论文和优秀解法有助于理解建模模式，但直接复用会带来拷贝风险，并削弱原创性。

#### 决策

Prior DB 检索只能产生结构化 prior cards。

Prior cards 可以影响模型路由和风险意识。

Prior cards 不得直接生成最终论文段落。

#### 推理

先验工作真正有价值的部分通常是：

```text
题型
模型族
适用条件
常见特征工程
常见图表类型
已知风险
基线方法
```

这些可以用卡片表示，而不复制表达或结构。

## ADR-006：最终论文自动化前必须使用论断绑定写作

#### 状态

```text
accepted
```

#### 背景

自动论文生成容易产生无支撑解释、夸大结论或不一致论断。

#### 决策

论文章节必须从已登记并绑定证据的论断生成。

段落应能追踪到：

```text
claim_id
evidence_id
result_id
figure_id
formula_id
citation_id
```

证据缺失时，系统应报告缺失证据，而不是写强论断。

## ADR-007：专家技能输出晋升前必须先归档为证据

#### 状态

```text
accepted
```

#### 日期

```text
2026-05-23
```

#### 背景

专家技能可以改进文献检索、引用检查、图表设计、章节写作、审稿回复、润色和终稿后展示。但不受控的技能输出可能绕过阶段、覆盖合同、改变冻结事实或引入无支撑论断。

#### 决策

使用默认拒绝的 Skill Router 策略和“先归档、后晋升”的输出模型。

每次技能调用必须先创建：

```text
input_manifest.yaml
raw_output.md
promotion_notes.md
```

只有当晋升说明给出有效目标、绑定、校验状态和必要人工决定后，技能输出才可以从建议变成下游可用内容。

---

## 决策：P1 运行时实现应从检查器开始，而不是执行器

Skill Router 运行时应首先实现 `scripts/check_skill_router.py`，作为策略校验、预检、归档脚手架、晋升校验和使用日志的确定性检查器。第一版不得直接执行专家技能。

## 决策：P1 Skill Router 运行时只作为防护栏生效

P1 路由器已实现为 `scripts/check_skill_router.py`。它可以校验策略、检查阶段-技能预检、初始化归档脚手架、校验晋升资格并追加使用日志。

它不得执行专家技能，不得直接修改合同，不得修改 `workflow_state.yaml`，不得修改提示词，也不得触碰最终论文产物。`scripts/check_gates.py` 通过 `skill_router_policy_gate` 检查路由器就绪状态，但不运行技能或晋升输出。

## 决策：P1.1 路由器加固先关闭晋升绕过问题

P1 路由器不得允许归档初始化绕过预检，晋升校验也不得信任脚手架文本或自报合同绑定。

立即生效：

- 归档脚手架需要成功预检授权。
- 脚手架式 `raw_output.md` 不可晋升。
- 合同绑定输出必须引用真实合同 CSV 中存在的 ID。

## 决策：P1.2 合同绑定必须证明关系一致性

路由器不得仅因多个 ID 各自存在，就将合同绑定晋升视为有效。写作、图表、图注和引用输出中，同一支持链中的 ID 必须关系一致。

立即生效：

- 空合同状态默认无效。
- `section_id` 是一等绑定令牌。
- 论断/证据和论断/结果对必须由同一有效绑定行或显式合同交叉引用支撑。
- 图表/结果对必须由 `figure_contract.csv` 或 `result_contract.csv` 的交叉引用字段支撑。
- 引用/论断和引用/章节对必须在提供 citation ID 时由 `citation_contract.csv` 支撑。

## 决策：学习层 v1 是 RAG 加轻量评分器，不是 LLM 微调

本地论文数据集可以用于构建建议型 prior cards、检索索引、拷贝风险检查和轻量工作流评分器。它不得用作直接答案库，也不得把历史论文文本复制到新提交中。

立即生效：

- `config/learning_policy.yaml` 是学习层事实来源。
- `13_prior_db/` 可以保存清单、抽象 prior cards、索引和拷贝风险报告。
- `16_learning/` 可以保存训练数据、本地评分器模型和建议型优化报告。
- `scripts/retrieve_prior_cards.py` 只能为 `prior_retrieval` 提供建议卡片。
- `scripts/route_weight_config.py` 可以记录 prior card ID 和置信度，但模型选择仍需要现有人工闸门。

## 决策：本地仪表盘是控制界面，不是工作流权威

本地 Web 仪表盘可以降低限时比赛中的操作成本，但不得成为第二套工作流引擎。

立即生效：

- `scripts/start_dashboard.py` 可以读取工作流状态和白名单产物用于展示。
- 仪表盘动作必须调用现有脚本，而不是直接编辑 `workflow_state.yaml`、合同、论文文件、结果文件或图表。
- 状态变更动作需要用户确认文本，并且只有绑定 `127.0.0.1` 时才启用。
- 同时只能运行一个仪表盘任务。
- UI 不得暴露 `stage all`、跨阶段执行、直接合同编辑、直接论文编辑、自动闸门确认或自动学习策略应用。
