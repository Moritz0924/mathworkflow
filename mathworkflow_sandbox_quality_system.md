# 1. 当前问题来源审查

## 1.1 问题概述

沙盒训练运行中已经暴露出两个核心问题：

1. **训练论文质量明显偏低**  
   论文能覆盖基本标题和部分结果，但篇幅、章节深度、模型推导、实验验证、图表数量与图表设计不足，更接近“技术初稿”而不是“可提交建模论文”。

2. **API Prompt Cache 命中率不稳定且偏低**  
   多轮调用中，Prompt 总 token 数持续增长，但缓存命中的 token 基本停留在固定的小范围，说明每一轮请求的大部分内容都在变化，无法稳定复用上下文前缀。

---

## 1.2 论文质量问题的直接证据

一次已完成的训练运行中：

- 训练状态标记为 `completed`；
- 合同校验通过；
- 修订队列已清空；
- 最终论文约为 **5,820 字符**；
- 初稿到第 5 轮修订的正文长度只从约 **5,722 字符**增长到约 **5,886 字符**；
- 迭代过程主要在修复合同、图表登记、修订队列、导出物，而不是补充模型、实验和论文论证。

因此，当前系统存在一个关键错位：

```text
流程完成、合同通过、队列清空
≠
论文达到可提交质量
```

---

## 1.3 根因总览

| 优先级 | 根因 | 直接后果 |
|---|---|---|
| P0 | 完成判定偏向流程合规，不是提交级论文质量 | 5,000～6,000 字符的短论文也可能被标记为完成 |
| P0 | `paper_full` 没有强制定义篇幅、章节深度、图表与实验下限 | 模型倾向于“覆盖标题”，而不是深入论证 |
| P0 | 图表校验主要检查文件与合同存在，不验证真实运行来源 | 静态 SVG、手写图、占位图可能混入合格产物 |
| P1 | 作者 Agent 与审稿/评分逻辑耦合 | 出现“自己写论文、自己打分、自己关闭问题”的偏差 |
| P1 | 修订循环优先处理 fail、major、合同和队列 | 论文深度、字数、实验质量不会被系统优先优化 |
| P1 | API 请求携带大量动态上下文 | Prompt Cache 只能命中少量固定前缀 |
| P2 | 未显式保障长文输出预算 | 长篇论文容易被 JSON 输出协议、默认上限或模型压缩策略截短 |
| P2 | System Prompt、Stage Prompt、Task Prompt 存在重复规则 | 输入 token 增长，但缓存复用和指令效果不成比例 |

---

## 1.4 根因一：完成条件没有区分“流程完成”和“质量通过”

当前训练闭环接近：

```text
阶段任务执行
→ 合同检查
→ 生成修订队列
→ 修订任务关闭
→ 校验通过
→ completed
```

更合理的闭环应是：

```text
阶段任务执行
→ 合同检查
→ 修订与导出
→ 独立质量审核
→ 质量门槛裁决
→ training_draft / submission_candidate / excellent_training_sample
```

当前系统能判断：

- 文件是否存在；
- 合同字段是否齐全；
- 图表是否登记；
- 修订任务是否关闭；
- 最终导出物是否存在。

但它尚不能可靠判断：

- 论文是否像正式建模论文；
- 每个问题是否被深入求解；
- 模型是否真正合理；
- 图表是否来自真实代码运行；
- 验证是否与题目类型匹配；
- 结论是否有足够证据支撑。

---

## 1.5 根因二：`paper_full` 的目标是“组装”，不是“达标生成”

现有 `paper_full` 阶段强调：

- 组装章节；
- 检查跨章节一致性；
- 暴露证据缺口；
- 检查图表、表格、公式、引用一致性；
- 不新增无证据事实。

这些原则是必要的，但并不等价于“生成一篇完整论文”。

缺失的硬约束包括：

- 最低有效正文长度；
- 各章节最低字数；
- 每个子问题必须包含的模型、结果、验证元素；
- 最低图表、表格、公式数量；
- 最低实验与对比数量；
- 论文必须达到的页面数或信息密度；
- 图表必须由当前运行脚本生成；
- PDF/DOCX 必须经过视觉审查。

结果是：模型很可能采用最省 token 的策略，用较短正文覆盖所有标题，从而满足“结构存在”的要求，却无法满足“提交级论证深度”。

---

## 1.6 根因三：修订队列不等于质量提升队列

当前迭代的优先级是：

```text
fail / major
→ 合同绑定
→ 缺失图表文件
→ 修订队列
→ 导出物
```

它并不天然包含：

```text
篇幅不足
→ 分章节补写

模型解释不足
→ 回到模型建立阶段补推导

验证不足
→ 回到结果冻结阶段补对照实验

图表叙事不足
→ 回到 figures 阶段补结果图、诊断图、敏感性图
```

因此，现有迭代更像是“把工件修到满足校验器”，而不是“把论文修到足够有说服力”。

---

## 1.7 根因四：图表数量、图表合同与图表真实性被混淆

当前图表风险主要有四类：

1. **同一图表可被多条合同记录重复引用**  
   如果统计逻辑按合同 CSV 行数计数，就可能把同一 `figure_id` 的多行绑定当作多张图。

2. **图表文件存在不等于图表来自真实计算**  
   静态 SVG、人工拼接图或手写数值图，只要文件存在，可能通过基础检查。

3. **脚本存在不等于脚本已经执行**  
   需要区分：
   - 有脚本；
   - 运行过脚本；
   - 脚本读取了当前结果；
   - 图表输出与当前结果一致。

4. **图表质量分不应由作者自己申报**  
   `quality_score` 必须有视觉审查或独立审核来源，不能仅由生成者填写。

应建立如下真实性链：

```text
result_file
→ execution_manifest
→ figure_script
→ figure_output
→ output_hash
→ figure_contract
→ paper_reference
```

链条任一环缺失，图表只能视为“草稿图”，不能用于提交候选或优秀训练样本。

---

## 1.8 根因五：作者与审核角色未充分隔离

现有自动审稿流程已经具备“只写审稿意见和修订任务、不直接改论文”的方向。

但仍存在风险：

```text
同一个 Agent 或同一套上下文
既生成论文
又生成评分
又判断问题是否关闭
```

这会导致：

- 自我偏袒；
- 为了通过而降低问题严重性；
- 把“建议”写成“已修复”；
- 用形式化合同掩盖论文深度问题；
- 审核意见重复作者自己的已知偏好。

因此，需要新增独立审核 Agent，并让规则引擎而非作者或审核 Agent 负责最终状态迁移。

---

## 1.9 根因六：缓存命中率低的结构性原因

### 1.9.1 请求结构本身是动态的

每个 API 请求大致由以下内容组成：

```text
System Prompt
→ 当前 stage
→ 当前 pending gate
→ Repository Rules
→ workflow_state.yaml
→ 当前 Stage Prompt
→ MCP Vision Observations
→ 当前 Controller Task
→ 当前 Workspace Context Snapshot
```

其中稳定部分通常只有：

```text
Repository Rules
+ 部分 workflow_state
+ 一小段固定系统规则
```

而以下内容在每轮、每阶段、每次修订后都会变化：

- `stage`；
- `pending_gate`；
- `call_id`；
- `run_id`；
- `workspace` 绝对路径；
- 当前修订队列；
- 当前合同内容；
- 当前论文草稿；
- 当前图表状态；
- 当前 Vision 观察结果；
- 当前任务描述；
- 当前 Workspace Context Snapshot。

因此，每次调用并不是“重复同一个 Prompt”，而是“带着不断变化的大型项目快照重新请求”。

---

### 1.9.2 缓存只能复用短前缀

在已有运行记录中，后续大请求的 Prompt token 常在 16k～25k 之间，但缓存命中 token 基本停留在约 3.1k～3.2k。

这意味着：

```text
稳定前缀约为 3k token
动态尾部约为 13k～22k token
```

因此缓存命中率会随动态上下文增长而下降。

---

### 1.9.3 Prompt 排列顺序不利于缓存

缓存通常依赖“内容相同且位置相同”的前缀。

当前结构中，过早出现动态字段会使共享前缀中断。例如：

```text
固定规则
→ 当前 stage 名
→ 当前 gate
→ 当前 stage Prompt
→ 当前动态任务
```

如果 stage 名、gate 或 stage Prompt 在固定规则之前或中间变化，就会使后续大量固定文本难以被复用。

---

## 1.10 缓存问题的治理方向

不改代码前，可先将原则固定为：

```text
稳定、长期、不随运行变化的内容
放在所有消息的最前部

动态、阶段化、运行时变化的内容
放在最后，并控制长度
```

推荐的逻辑结构：

```text
[Stable System Prefix]
- 审核/写作通用角色规则
- JSON 协议
- 永久禁止动作
- 固定仓库治理原则

[Stable Project Prefix]
- 项目全局约束摘要
- 合同字段定义摘要
- 质量量表摘要

[Stage Delta]
- 当前 stage 的少量差异化规则

[Run Delta]
- 当前任务
- 当前修订队列
- 当前文件差异
- 当前结果摘要
```

核心原则：

- 不重复注入完整大文件；
- 不把绝对路径、时间戳、随机 run id 放在稳定前缀；
- 不把整个论文和整个合同反复塞入 Prompt；
- 用摘要、索引、差异包代替全量快照；
- 使用单独的稳定 Prompt 版本号与 Context Snapshot Hash。

# 2. 训练论文质量验收量表

## 2.1 适用范围与总原则

本量表用于判断沙盒训练产物的论文质量，不用于判断“工作流是否跑完”。

默认按 C2 型常规数学建模题设置，覆盖七个维度：

| 维度 | 总分 | 机器检查 | 人工/审核 Agent 检查 |
|---|---:|---:|---:|
| 篇幅与有效信息密度 | 10 | 6 | 4 |
| 章节深度与论证闭环 | 18 | 6 | 12 |
| 模型推导与方法合理性 | 18 | 3 | 15 |
| 实验验证与结论可信度 | 18 | 7 | 11 |
| 图表真实性与可复现性 | 16 | 10 | 6 |
| 图表密度与叙事覆盖 | 10 | 6 | 4 |
| 论文格式与交付质量 | 10 | 6 | 4 |
| **总计** | **100** | **44** | **56** |

### 总体规则

1. **红线优先于总分**：出现伪造、无来源结论、图表不真实等红线，直接拒绝。
2. **机器检查负责一致性与可追溯性**：文件、编号、Hash、合同、导出与执行记录。
3. **审核 Agent 负责方法与表达质量**：模型是否合理、验证是否可信、图表是否真正服务论证。
4. **高分维度不能掩盖低分核心维度**：模型、实验、图表真实性必须分别达标。

---

## 2.2 维度一：篇幅与有效信息密度（10 分）

### 默认目标

| 指标 | C2 默认门槛 |
|---|---:|
| 正文有效中文字符数 | 16,000–24,000 |
| 摘要 | 400–800 字 |
| 问题分析 | ≥1,000 字 |
| 每个核心子问题的建模、求解与结果 | ≥1,200 字 |
| 验证与敏感性分析 | ≥1,000 字 |
| 结论、局限与建议 | ≥600 字 |

“有效正文”不包括：参考文献、代码、日志、原始数据附录、大量重复文本和纯表格堆砌。

### 机器检查（6 分）

| 检查项 | 满分条件 |
|---|---|
| 总有效字数 | 达到当前档位要求 |
| 章节字数下限 | 摘要、分析、模型、结果、验证、结论均达标 |
| 重复率 | 高重复段落、模板句和重复结论比例低于 10% |
| 空洞段落 | 不存在连续两段以上无结果、无公式、无解释的空泛叙述 |
| 草稿一致性 | `full_draft.md` 与最终提交稿无异常缩短或内容丢失 |

### 审核 Agent 检查（4 分）

| 得分 | 判断标准 |
|---:|---|
| 4 | 内容充实，无明显凑字；每段承担解释、推导、比较或结论功能 |
| 3 | 基本完整，少量泛化描述或局部展开不足 |
| 2 | 结构存在，但多数章节偏短、偏概述 |
| 1–0 | 明显初稿化、拼接化或依赖套话填充 |

### 红线

- 正文少于 10,000 有效字符；
- 主要章节只有标题、表格或两三段泛化文字；
- 大量重复“模型具有良好效果”“具有较高鲁棒性”等空泛句式。

---

## 2.3 维度二：章节深度与论证闭环（18 分）

### 必需章节

1. 摘要与关键词；
2. 问题重述与问题分析；
3. 模型假设、符号说明与数据说明；
4. 子问题一：模型建立、求解与结果；
5. 子问题二：模型建立、求解与结果；
6. 子问题三：模型建立、求解与结果；
7. 模型检验、敏感性、稳健性或不确定性分析；
8. 模型优缺点与改进方向；
9. 结论与应用建议；
10. 参考文献、附录与代码说明。

### 机器检查（6 分）

| 检查项 | 满分条件 |
|---|---|
| 必需章节覆盖 | 10 个章节或等价结构完整存在 |
| 子问题覆盖 | 每个题目均存在“问题→模型→求解→结果→验证”链条 |
| 标题层级 | 一级、二级、三级标题层级连续，无跳级 |
| 章节最小深度 | 核心章节至少含正文、公式/表格/图、结论中的两类以上 |
| 引用一致性 | 图、表、公式、子问题编号均可定位且不冲突 |
| 缺口暴露 | 未解决问题必须列入风险或局限章节 |

### 审核 Agent 检查（12 分）

| 评审维度 | 分值 | 重点 |
|---|---:|---|
| 问题拆解合理性 | 3 | 子问题是否递进，是否避免重复建模 |
| 论证主线 | 3 | 从题意到模型、结果、结论是否自然闭环 |
| 章节衔接 | 2 | 前后章节是否引用彼此结果，而不是孤立拼接 |
| 结论对应性 | 2 | 每个结论是否能回指到模型或实验结果 |
| 局限性表达 | 2 | 是否明确适用范围、假设代价与潜在失效情形 |

### 红线

- 某个子问题只有结果，没有模型或解释；
- 结论中出现正文从未证明的新观点；
- 验证章节与前文模型、数据、结果无关联。

---

## 2.4 维度三：模型推导与方法合理性（18 分）

### 机器检查（3 分）

| 检查项 | 满分条件 |
|---|---|
| 模型合同 | 每个核心模型都有 `model_id`、公式、变量、代码路径、结果引用 |
| 公式合同 | 核心公式都有 `formula_id`，正文引用一致，符号有定义 |
| 结果绑定 | 每个重要结论至少绑定一个 `result_id`、`figure_id` 或可验证数据来源 |

### 审核 Agent 检查（15 分）

| 评审维度 | 分值 | 重点 |
|---|---:|---|
| 建模选择合理性 | 4 | 是否符合题目目标、数据规模、约束和业务语境 |
| 假设合理性 | 3 | 假设是否必要、可解释，是否说明其影响 |
| 数学表达与推导 | 4 | 目标函数、约束、变量、算法逻辑是否完整且正确 |
| 方法比较与选型 | 2 | 是否有候选模型比较，而非直接指定一个方法 |
| 创新与适配性 | 2 | 是否体现针对题目的改造、组合或解释，而非模板套用 |

### 评分锚点

| 得分范围 | 特征 |
|---|---|
| 13–15 | 模型定义严谨、推导完整、方法选择有明确证据支撑 |
| 9–12 | 模型基本合理，但个别假设、算法或推导展开不足 |
| 5–8 | 有模型名称和公式，但核心逻辑解释较浅 |
| 0–4 | 仅罗列方法名，无法说明为什么能解决问题 |

### 红线

- 公式没有变量含义；
- 优化模型缺少目标函数或关键约束；
- 预测模型没有特征、目标、训练方式或评价指标；
- 使用复杂模型但没有解释其必要性。

---

## 2.5 维度四：实验验证与结论可信度（18 分）

### 机器检查（7 分）

| 任务类型 | 最低验证要求 |
|---|---|
| 预测/回归 | 基线模型、交叉验证或测试集、误差指标、残差或不确定性分析 |
| 分类/评价 | 基线、混淆矩阵或核心评价指标、阈值或类别不平衡说明 |
| 优化/调度 | 可行性检查、基线方案比较、约束满足证明、参数敏感性 |

# 4. Sandbox Quality Auditor 审核 Agent 设计

## 4.1 定位

新增一个专门的 **Sandbox Quality Auditor（SQA）**，替代沙盒训练中的“模拟人工审核员”。

SQA 不是论文作者，也不是修订执行者，而是独立的质量裁判：

- 审查论文是否达到“训练初稿 / 提交候选 / 优秀训练样本”；
- 生成有证据定位的评分与修订任务；
- 自动决定沙盒论文是否通过质量闸门；
- 不修改论文、代码、结果、图表和正式工作流状态；
- 不替代正式项目中的人类最终决策。

---

## 4.2 权限边界

| 项目 | SQA 权限 |
|---|---|
| 读取论文、结果、合同、图表、导出物 | 允许 |
| 读取执行记录、图表脚本、运行日志 | 允许 |
| 渲染 PDF/DOCX 页面并进行视觉审查 | 允许 |
| 写入评分表、质量报告、修订任务 | 允许 |
| 修改论文、代码、数据、图表、合同 | 禁止 |
| 标记修订任务为已关闭 | 禁止 |
| 修改 `workflow_state.yaml` | 禁止 |
| 让正式工作流跳过人工闸门 | 禁止 |
| 推荐沙盒质量档位 | 允许 |
| 最终状态迁移 | 由规则引擎执行，不由 SQA 直接执行 |

核心原则：

```text
作者 Agent 负责产出
修订 Agent 负责修复
审核 Agent 负责否决与评分
规则引擎负责最终状态迁移
```

---

## 4.3 在流程中的位置

建议将现有 `auto_review` 调整为“作者侧自检”，不再作为最终质量结论来源。

```text
paper_full
   ↓
auto_review
作者自检：发现明显合同、结构、引用问题
   ↓
revision
作者修订
   ↓
compile / final_export
生成 Markdown、DOCX、PDF、渲染页
   ↓
quality_audit
SQA 独立审核：评分、证据核验、视觉审查、任务下发
   ↓
┌──────────────────┬────────────────────────┐
│ 通过质量闸门      │ 未通过质量闸门           │
│                  │                          │
│ 输出质量档位      │ revision → quality_reaudit │
└──────────────────┴────────────────────────┘
```

`quality_audit` 是沙盒训练的硬质量闸门。  
`quality_reaudit` 使用同一量表，但只核验上一轮修订任务是否真正解决。

---

## 4.4 独立性设计

### 最低实现标准

即使仍使用同一个模型供应商，也必须满足：

- 独立 API 调用；
- 独立 System Prompt；
- 不传入作者 Agent 的思维过程、草稿生成指令和自评分数；
- 审核输入来自冻结快照；
- 审核 Agent 只能看到最终产物、合同、执行证据和量表；
- 审核 Agent 不知道论文是第几轮修订；
- 审核 Agent 不得读取旧评分，除非处于 `quality_reaudit` 模式。

### 推荐标准

使用不同的审核模型，或至少不同的审核配置：

| 角色 | 建议配置 |
|---|---|
| 作者 Agent | 深度推理、允许生成文件动作 |
| 修订 Agent | 深度推理、仅处理修订任务 |
| SQA | 低温度、严格 JSON、只读、证据优先 |
| 视觉审核器 | 图像/PDF 渲染输入，检查版式与图表 |
| 仲裁审核器 | 仅在双审核分差较大时介入 |

同模型不同 Prompt 可以降低自我偏袒，但不能完全消除同源偏差；不同模型或双模型交叉审核更可靠。

---

## 4.5 五个审计面

SQA 是一个专门 Agent，但内部应按五个固定审计面输出结果。

| 审计面 | 核心问题 | 主要证据 |
|---|---|---|
| 证据与可追溯性 | 结论、结果、图表是否可回溯 | result / claim / figure / formula contracts |
| 模型合理性 | 模型是否真正解决题目 | 问题分析、假设、公式、代码、模型说明 |
| 实验与验证 | 是否有基线、对比、敏感性或误差分析 | 结果文件、实验表、验证图、运行日志 |
| 图表与视觉 | 图是否真实、清晰、服务论证 | 图文件、脚本、生成记录、PDF 渲染页 |
| 论文与评委视角 | 结构、篇幅、连贯性、局限、提交感 | 全文、章节、摘要、结论、导出物 |

## 4.8 自动判定逻辑

```text
先检查红线
    ↓
检查机器硬门槛
    ↓
执行 SQA 七维评分
    ↓
检查每个核心维度最低分
    ↓
由规则引擎输出质量档位
```

### 红线

出现任一项，直接标记 `quality_rejected`：

- 核心结论无结果来源；
- 图表伪造、手写静态图冒充真实实验图；
- 图表、正文、结果文件数据冲突；
- 复制风险报告为 fail；
- 核心子问题没有模型或没有结果；
- 论文存在 TODO、占位文本、待补内容；
- 没有可用的最终 PDF/DOCX；
- 审核 Agent 无法定位关键结论证据。

### 档位裁决

| 档位 | 必要条件 |
|---|---|
| 训练初稿 | 总分 ≥75；无红线；核心维度达到初稿最低分；允许 minor 与有限 major 待修 |
| 提交候选 | 总分 ≥85；无 fail/major；所有图表真实可追溯；DOCX/PDF 视觉检查通过 |
| 优秀训练样本 | 总分 ≥90；无 fail/major；模型、验证、图表真实性均达到高档；全部修订任务关闭 |
| 需要修订 | 无红线，但不满足当前档位门槛 |
| 拒绝 | 有红线或多轮修订无改进 |

## 4.9 SQA 关键 Prompt 规则

```text
你是 MathWorkflow Training Sandbox 的独立质量审核 Agent。

你的职责是审查，不是创作，不是润色，不是修订。

严格规则：
1. 只基于 audit_snapshot 中可见证据评分。
2. 未观察到证据时，必须标记为“证据不足”，不得自行推断通过。
3. 每个 fail / major 必须给出 target_artifact、target_location、evidence_refs、
   required_action 和 acceptance_check。
4. 不得修改论文、结果、图表、代码、合同、工作流状态。
5. 不得关闭修订任务，不得宣称问题已修复。
6. 图表真实性必须检查：图文件、结果来源、脚本来源、执行证据四者。
7. 审核输出只能为严格 JSON 文件动作。
8. 最终只给出 recommended_decision；质量档位由规则引擎决定。
```

---

## 4.10 防止“自评通过”的机制

| 风险 | 控制措施 |
|---|---|
| 作者自评偏高 | SQA 与作者独立会话、独立 Prompt、冻结快照 |
| 修订任务假关闭 | 只有规则引擎能关闭；作者只能标记 `claimed_resolved` |
| 用合同刷通过 | 图表与结果必须有脚本和执行证据，不只检查文件存在 |
| 只修格式不修内容 | 每个 major 必须含语义或执行层面的验收条件 |
| 反复改写却没有进步 | 连续两轮总分无提升或 major 不减少，标记 `quality_stalled` |
| 视觉质量靠自报分数 | 使用 PDF/DOCX 渲染页进行独立视觉审核 |
| 评分不稳定 | 同一冻结快照可采用双审计；分差大于 8 分时触发仲裁 |

# 5. 推荐状态与决策逻辑

## 5.1 状态机

```text
draft_ready
    ↓
author_self_review
    ↓
revision_in_progress
    ↓
export_ready
    ↓
quality_audit_pending
    ↓
quality_audit_running
    ├── quality_rejected
    ├── quality_needs_revision
    │       ↓
    │   revision_in_progress
    │       ↓
    │   quality_reaudit_pending
    ├── quality_training_draft
    ├── quality_submission_candidate
    └── quality_excellent_training_asset
```

正式工作流只读取最终状态与审计报告摘要，不接受 SQA 直接修改正式项目状态。

---

## 5.2 推荐裁决算法

```text
1. 运行静态硬门槛检查
2. 创建 audit_snapshot
3. 运行 SQA 评分
4. 检查 redline_count
5. 检查总分、机器分、审核分
6. 检查七维最低分
7. 检查 open_fail/open_major
8. 输出 recommended_decision
9. 由规则引擎写入质量状态
```

伪代码：

```python
if redline_count > 0:
    verdict = "quality_rejected"
elif not hard_gate_passed:
    verdict = "quality_needs_revision"
elif total_score >= 90 and all_core_dimensions_high and no_open_major:
    verdict = "quality_excellent_training_asset"
elif total_score >= 85 and all_submission_thresholds and no_open_major:
    verdict = "quality_submission_candidate"
elif total_score >= 75 and draft_thresholds_met:
    verdict = "quality_training_draft"
else:
    verdict = "quality_needs_revision"
```

# 7. 结论

当前沙盒训练系统已经具备：

- 阶段化运行；
- 合同约束；
- 修订队列；
- 图表与结果登记；
- 导出与基础校验；
- 自动审稿雏形。

但要把它从“能完成工作流的训练脚本”升级为“能稳定产出高质量建模论文的训练系统”，必须建立：

```text
质量量表
+ 三级档位
+ 图表真实性链
+ 独立审核 Agent
+ 审核后复审机制
+ 规则引擎裁决
```

最终目标不是让 Agent “自己宣布完成”，而是让每一篇训练论文都能回答：

```text
是否完整？
是否真实？
是否可复现？
是否有足够验证？
是否达到对应质量档位？
为什么？
```
